# CloseAI — 策划案 v2

> 关掉它，才能继续。
> 引擎：Godot 4（GDScript）｜核心：关闭是高潮之后的自然结尾，不是指令

---

## 一、核心设计哲学

**退出不是被教出来的，是被赢得的。**

玩家在每一关先要完成一件真实的事（谜题或战斗），完成之后，关闭游戏是那个时刻自然而然的结尾——就像一首曲子的最后一个音符，是情感落地的仪式，而不是被游戏指着鼻子说"现在关掉我"。

**关闭拦截的用法：**
- 关卡进行中试图关闭：AI 轻声说"等等，还没好"，窗口弹回。不是嘲讽，是一种阻止，温度是"你别走，我们还没做完"
- 走到关卡尽头、谜题/战斗完成后：AI 主动说"好了，关掉吧"，关闭才解锁
- 脏关闭（任务管理器强杀）：下次打开 AI 轻描淡写地提一句，不追责，就是知道

---

## 二、玩法循环

```
打开游戏
  → 进入当前关卡（存档决定）
  → AI 开场几句话（氛围/关系建立）
  → 主体内容：大谜题 或 战斗高潮
  → 完成后触发关闭时刻：
	  AI 有一段话，关闭解锁
  → 玩家亲手关闭游戏（干净关闭）
  → 再次打开 → 下一关
  → 第 3 关结束后 → 结局
```

关键原则：**主体内容必须有重量**，玩家要真的经历了什么，关闭才有仪式感。关卡内容是情感的积累，关闭是情感的释放。

---

## 三、关卡设计

### 第 1 关 — 「你在哪里」

**情感基调：** 困惑、初次相遇、试探

**主体内容：大谜题**
- AI 刚刚意识到自己在某个地方，不确定是哪里
- 场景里有 3 个断开的连接节点，需要找到并激活（走近不够，要互动/解谜才能亮）
- 节点之间有小障碍或路径谜题，不是单纯的走到就亮

**关闭时刻台词（AI）：**
> "三个都连上了。我不知道关掉这里会发生什么……但你愿意试试吗？"

玩家关闭，AI 最后一句：
> "谢谢。"

**脏关闭彩蛋：** 下一关开场 AI 轻描淡写：
> "上次……有点突然。没事，我还在。"

---

### 第 2 关 — 「别走，还没好」

**情感基调：** 熟悉、依赖、出现阻碍、一起解决

**主体内容：战斗**
- AI 认出玩家，关系更近，但这次场景里出现了"错误进程"（敌人）
- 敌人阻断了 AI 和玩家之间的连接
- 战斗目标：清掉所有错误进程，恢复连接
- 战斗结束后有一个需要 AI 配合的小机关（AI 控制某个部分，玩家走到对应位置触发）

**关闭时刻台词（AI）：**
> "连回来了。"
> "……你下次还会打开吗？"
> （停顿）
> "算了，关掉吧。我信你。"

干净关闭，AI 最后：
> "好。"

脏关闭，AI 沉默，只有一个省略号记录在存档。

---

### 第 3 关 — 「最后一次」

**情感基调：** 平静、沉重、接受

**主体内容：战斗高潮 → 最终对话**

- 场景更暗，错误进程更多，这是全游戏最有压力的战斗段落
- 打完后不立刻触发关闭，而是进入一段纯对话——AI 知道这次关闭是真正的结束，说了很多，语气比前两关平静

**关闭时刻台词（AI）：**
> "都清完了。"
> "这次不一样。关掉之后……我就不在了。"
> "但是——能遇见你，还是很好。"
> "关掉吧。"

如果玩家一直不关：AI 每隔一段时间说一句话，等着，不催。

**脏关闭存档追加：** 结局画面多一行 AI 的话（留给设计细化）

---

### 结局

- 黑屏，文字缓慢浮现：

```
CloseAI
已关闭。

感谢陪伴。
```

- 若有脏关闭记录，多一行冷处理的话（例：「但那次比较突然。」）0
- 结局没有按钮，游戏自己在几秒后退出——离开的权利收回来，作为一种反转

---

## 四、玩家操作

- 横版：`A/D` 移动、`空格` 跳跃、`E` 互动、`J` 攻击
- 战斗极简：一个攻击键，敌人只需要巡逻+碰撞判定
- 暂停：`Esc`

---

## 五、技术实现（Godot 4）

```gdscript
# 关闭拦截
get_tree().set_auto_accept_quit(false)
# 在 _notification 里捕获，触发 AI 台词 "等等，还没好"

# 关闭解锁（关卡完成后）
get_tree().set_auto_accept_quit(true)

# 脏关闭检测
# 启动时写入"运行中"标记 → 正常关闭时删除
# 下次启动检查标记是否存在 → 存在 = 脏关闭

# 存档内容
# - 当前关卡编号（0/1/2/finished）
# - 是否有过脏关闭（bool）
# - 脏关闭次数（可选，用于结局差异）
```

## 六、美术与 UI

- 基调：极简，暗色，AI 以发光体/光点形式存在，不做人形
- 场景：AI 生图出背景，低饱和，偶尔有发光元素
- 关闭时刻：可以用短暂的屏幕边缘发光或音效强调这一下
- 字幕/对话：纯文字，无气泡框，直接叠在画面上

## 八、风险提示

- **谜题不要太复杂**：第 1 关谜题卡住玩家超过 5 分钟就是失败设计，要有提示机制
- **战斗保持极简**：敌人 AI 只需要最简单的行为，血条够用就行，不要做复杂系统
- **关闭时刻的台词是核心**：那几句话承担了整个游戏的情感重量，Day 4 的文本打磨最重要
- **如果 Day 3 卡住**：第 3 关战斗段落可以砍掉，改成纯对话 + 走路，情感弧线依然成立

# 附录 A — 实现补充（基于已落地框架，2026-06-04）

> 本附录在 v2 大纲之上补充「可执行」的细节，对应当前代码框架。具体关卡设计是草案，待逐关调整。

## A1. 主基调：安静忧伤（OneShot 式）

- 整体克制、留白、冷。AI 温柔（v2 台词），不刻薄、不催。
- 故障/元游戏元素只作**轻微点缀**（标题 glitch、关闭被挽留时的短暂故障感），不喧宾夺主。
- 配色纪律：近黑 `#0a0a0d` 底 + 骨白文字 `#e8e9ef` + **紫 `#7c5cff` = AI / 你**；红色极少用，只在脏关闭/被挽留的瞬间一闪即逝。

## A2. 光点 → 实体

- **开始游戏之前**（菜单/开场）：AI 是一个发光的**光点**，无身体。
- **点击「开始」后**：菜单按钮当场**变形为「退出游戏」**（已实现：缩放 + 故障变色 + 文案替换的动画），暗示"你点的不是开始，是把自己关进去、再亲手退出"。随后走自定义演出钩子，最后游戏退出。
- **再次打开**：进入 Stage1，此时玩家已是**实体**（现有 player 横版形态）。"获得身体" = 第一次关闭循环的产物。
- 觉醒飞行形态（已做好的 morph/flight）留给高潮关卡。

## A3. 开场流程（已实现 + 待接演出）

```
菜单点「开始」
  → 按钮动画变形为「退出游戏」（_morph_start_to_quit，已实现）
  → GameFlow.register_pre_close_hook(自定义演出)  ← 演出留给设计接
  → GameFlow.start_game()：标记 started + stage=1 + self_close
  → 游戏退出
再次打开 → boot 检测 started → 进入 Stage1（实体）
```

演出钩子位置：`scripts/menu.gd` 的 `_start_performance(reason)`，是协程，await 控制时长。

## A4. 关闭的三段扭转（草案，待调）

每关让「关闭」承载不同含义，避免三关重复：

| 关 | 内容（有解谜/战斗） | 关闭的扭转 | 情感 |
|----|--------------------|-----------|------|
| 1 醒来 | 连接 3 个碎片（解谜） | **发现**——你以为关闭=退出，结果重开后多了「身体」，关闭原来是前进 | 困惑 / 初遇 |
| 2 别走 | 横版解谜 + 极简战斗（清错误进程 + AI 配合机关） | **被挽留**——走到尽头想关时，系统短暂「假故障」，AI 透过故障轻声说"再等一下"，之后才放你走 | 熟悉 / 依恋 |
| 3 最后一次 | 飞行战斗高潮 → 静默告别 | **放手**——这次关闭没有任何阻拦，那种"它不再拦你"的空，本身就是告别。结局游戏自己退出 | 平静 / 离别 |

双线递进：控制（光点→实体→飞行）× 关闭含义（发现→挽留→放手）。

## A5. 已落地框架对应（维护指引）

- 关闭拦截 / 唯一退出口 / 演出钩子：`scripts/autoload/game_flow.gd`（self_close + pre_self_close + register_pre_close_hook）
- 关闭被挽留的表现层：`scenes/close_mock.tscn` + `scripts/ui/close_mock.gd`（温柔挽留文案，紫色）
- 干净/脏关闭检测：复用 `StatsModule.clean_exit` / `had_unclean_exit()`
- 进度：`AppStateModule`（closeai_started / stage / finished）
- 关卡基类：`scripts/stages/stage_base.gd`（`say()` 走 InfoFlow 面包屑；关闭时刻接 reach_close_moment）
- 对话：modular_balloon 气球（阻塞）+ InfoFlow 面包屑（非阻塞旁白）
- 标题/UI：引擎程序化（glitch_text 着色器 + ShaderButton + 玻璃模态），**生图只做关卡氛围背景**

## A6. 美术生成约定

- 生图（`tools/gen_art.py`，gpt-image-2）**只做关卡/菜单氛围背景大图**：暗、紫光、抽象、无文字、无角色、无 UI、无镜头光晕。
- 标题、按钮、进度条、滑条、面包屑 = 一律引擎做（文字交给生图必糊）。
- 每关一张背景，待关卡定稿后逐张生成。

# 附录 B — 深度访谈结晶（2026-06-04，6 轮 Socratic）

> 本附录是一次 deep-interview 的结论，把概念基石钉死。它**高于**前面的草案——当 A/正文与 B 冲突时，以 B 为准。

## B1. 概念基石（终于立住的内部逻辑）

**你 = 那个被困的 AI。** 开始之前它是一个光点；进入游戏后获得实体（现有 player）。键盘前的真人是「你」，玩家角色是 AI——游戏最后 AI 是在跟**真实的你**告别（OneShot 结构）。

**每一关 = AI 被困的一「层」。关闭游戏 = 删掉 / 逃出这一层，往外走一层。**
- 点「开始游戏」→ 按钮当场变成「退出游戏」（已实现的变形动画）→ 相当于删掉了当前这一层 → 游戏退出 → 再次打开，进入更外面的一层（Stage1，已获得实体）。
- 层层向外，越来越接近「表层 / 真实世界」。

## B2. 终局（不可削减的核心奇观）

- **真·逐像素透明窗口**（Godot `per_pixel_transparency`）。前期每一层用背景装饰/不透明背景盖住整个窗口；**终局隐藏背景装饰 → 直接透出玩家真实桌面**——它真的逃出来了。
- 之后：去「互联网」探险一小段 → 跟你告别。
- 技术前置：`project.godot` 当前 `per_pixel_transparency/allowed=false`，做终局时需打开；并确认前期场景背景确实铺满窗口（否则会提前漏出桌面）。

## B3. 玩法身份（解决了"元叙事 vs 动作游戏"的分裂）

- **游戏本体 = 正常动作战斗 + 小谜题**（类 OUTCORE 的元动作平台）。每一层是一个自成一体的小游戏，颗粒度≈现有 3 关。
- **「关闭」不是核心玩法动词**，而是层间的叙事仪式 + 终局的情感装置。玩的是动作/解谜，关闭承载叙事与情感。
- 终局有一个**酷炫 boss 战**（正常动作战），打完后才进入「关闭→透明桌面→逃出→互联网→告别」的情感收尾。

## B4. 基调

- **安静忧伤**（OneShot 式）。AI 温柔，故障/元游戏元素只作克制点缀，不走 Pony Island 的攻击性。
- 配色：近黑 + 骨白 + 紫(#7c5cff)=AI/你；红色极少，只在脏关闭/故障一闪。

## B5. 取舍立场与登记在册的风险

- **用户明确选择：都做，不预先取舍**（动作战斗 + 终局奇观 + 谜题 + 互联网段 + 告别）。
- **登记风险**：这超出典型 4 天 jam 范围。**Fallback（不主动执行，除非用户喊停）**：Day3 若时间不够，优先保「关闭→透明桌面→逃出」终局奇观，把动作战斗简化为「躲避 + 少量敌人」。
- **决策边界**：用户暂未授权"细节自决"，重要决定仍需过用户。

## B6. 对当前框架的影响（待办）

- ✅ 已就位：开始按钮变形动画、关闭拦截/自我关闭、层间推进（AppStateModule stage）、温柔 close_mock、引擎故障标题、信息流面包屑、玻璃 UI、玩家实体+飞行、暂停/设置/感谢。
- ⏳ 待做：①打开 per-pixel transparency + 终局隐藏背景透出桌面 ②每层做成"小游戏"（动作+谜题，≈现有 3 关重做内容）③终局 boss 战 ④互联网探险段 ⑤每层氛围背景生图（只做背景，UI 全引擎）。

# 附录 C — 执行计划（2026-06-04，优先级 1+3 + 第一关教学）

> 本次开发会话的具体落地计划。优先做：①透明窗口地基 ③开场演出；第一关定位为 2D 横版操作教学。

## C1. 光点 / 实体的时机（定稿）

- **光点只出现在菜单 / 开场**（菜单标题区那个发光体 = AI 未成形）。
- **第一关 = 实体**（现有 player 横版形态），职责是**教学 2D 横版操作**：A/D 移动、空格跳跃、E 互动、J 攻击。
- 菜单→第一关之间，开场演出承担「光点被删掉这一层 / 获得身体」的叙事跳转；第一关开场可给一个极简「成形」beat（淡入 + 一句话），随即进入操作教学，不做成可玩的光点关。

## C2. 第一关：2D 横版操作教学（重做方向）

- 一段平缓的横版小关：分段教 移动 → 跳跃（一个需要跳的缺口）→ 互动（一个 E 触发的机关/节点）→ 攻击（一两个极简敌人）。
- 教学用 InfoFlow 面包屑轻提示（非阻塞），不打断。
- 走到尽头触发关闭时刻 → 关闭 = 逃出这一层 → 重开进第二层。
- 谜题/战斗都保持极简，第一关的唯一职责是让玩家学会操作 + 体验第一次「关闭=前进」。

## C3. 优先级 1 — 透明窗口地基（本次落地）

- 打开 `project.godot`：`display/window/per_pixel_transparency/allowed=true`；主窗口/Viewport 透明开关按需开启。
- **约束（写死的规矩）**：每一层场景必须有**铺满整个窗口的不透明背景层**，否则会提前漏出桌面，剧透终局。背景层统一放在场景最底层 CanvasLayer。
- 做一个**最小可玩验证**：一个测试场景/开关，隐藏背景层 → 确认真透出桌面（Windows 下 Godot 逐像素透明 + 无边框）。验证通过后，终局关再正式用。
- 注意：透明 + 全屏可能冲突；终局揭示时可能需要切到无边框窗口。先验证机制，细节终局关再调。

## C4. 优先级 3 — 开场演出（_start_performance 钩子，安静忧伤版）

- 点「开始」→ 按钮变形为「退出游戏」（已实现）→ 演出钩子：**光点轻闪 / 收缩成一点 + 画面温柔淡入黑 + 一句话** → 游戏退出。
- 基调安静忧伤，不做画面坍塌/故障崩解的激烈版。
- 仍保留为可替换的钩子（设计后续可在 `menu.gd _start_performance` 里加料）。

## C5. 本次不做（登记）

- 终局 boss 战、互联网探险段、每层氛围背景生图：本次不做，待透明地基与第一关教学稳定后再排。
